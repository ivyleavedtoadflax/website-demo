Managing this site with GitHub

A plain-English guide to how this website is built and changed — written for people who don't use GitHub day to day.

The big idea

The website has two states:

Live

What the public sees. It only changes when someone approves a change.

Drafts

Proposed changes that are not yet live. Anyone can work on one with no risk to the real site.

A draft only becomes live after a person reviews and approves it.

How a change gets made

You never need to touch code or know the technical steps:

  1. You describe the change — for example, "add a page about stakeholder mapping" or "make the homepage intro simpler". AI does the editing.
  2. A draft is created, separate from the live site, so nothing the public sees changes yet.
  3. You get a preview link — the draft is published to its own temporary web address, so you see it as a real page, not as code.
  4. Someone reviews it — a reviewer reads the change, asks for tweaks, or approves.
  5. It goes live — once approved, the change publishes automatically and the temporary preview is removed.
In one line: describe → draft → preview → review → approve → live. The technical steps are handled for you.

Why it's done this way

  • The live site can't break by accident — drafts stay separate until approved.
  • Every change is visible — you can always see exactly what changed and who changed it.
  • Anything can be undone — every change is recorded, so mistakes are easy to roll back.
  • Automatic checks can catch common errors, like a broken link, before a change goes live.

The one role worth having

A single person who's comfortable reviewing changes — a content lead, say — is enough to keep quality high. They don't write code; they read the preview, check it reads well, and approve. That's the safeguard against anything odd slipping through.
